# データライフサイクル管理機能 拡張仕様書（wasm_runner）

> **改訂（2026-07-10）**: Ed25519 署名認証・WASM 事前アップロード・データごとの
> 実行可能プログラムホワイトリスト（`docs/260710-signature-auth-and-program-upload-workplan.md`
> の U1–U5）の導入に伴い、認証・信頼モデルに関わる §2・§3・§5・§7・§8・§10・§11 を
> 改訂した。とくに「利用者＝データオーナー本人」という当初前提は撤廃され、
> **アップローダ（データ提供者）とランナーオーナー（実行者）は別主体になり得る**。
> この前提変更への対応はデータごとのホワイトリスト（§7.2）による。

## 0. 文書の位置づけ

本書は `wasm_runner`（TEE = Gramine-SGX 上で動作する WASM 実行サーバー）に、
データの**登録・利用・削除のライフサイクル管理**機能を追加するための設計仕様書である。
現行実装（`wasm_runner/main.go`）は「受け取った WASM バイナリをその場で実行して
結果を返すだけ」のステートレスなサーバーであり、データという概念自体を持たない。
本拡張により、`wasm_runner` は次の2つの役割を明確に分離して担う。

| 役割 | 担当 |
|---|---|
| データのライフサイクル管理（登録・利用・削除、削除証明の発行） | `wasm_runner`（ホスト側 Go プログラム） |
| ライフサイクル内で行う任意の計算処理 | 利用者が提出する WASM プログラム |

最終目的は、「登録されたデータが、いつ・誰によっても複製されず、最終的に削除された」
ことを第三者が検証可能な形で証明できるようにすること（**削除証明**）。

## 1. 背景・目的

- データの削除証明を成立させるには、データがシステム内のどこに存在し、どんな処理を
  経て、いつ消えたかという**ライフサイクル全体**を管理下に置く必要がある。
- ライフサイクルの管理（状態遷移・アクセス制御・削除の実施・証明の発行）は
  `wasm_runner` が TEE 内で行う。TEE を信頼の起点とすることで、インフラ管理者を
  含む外部者による干渉・盗聴からライフサイクル管理そのものを保護する。
- ライフサイクル内で実行される処理（データの集計・変換・学習など）は、利用者が
  自由に持ち込む WASM プログラムに任せる。ただし、その WASM プログラムが
  **データを外部に送信できてしまうと、複製が生成され削除証明が意味を失う**ため、
  外部送信を強制的に禁止する必要がある。
- 本書はこの要件を満たすための `wasm_runner` の拡張仕様を定義する。

## 2. 用語

| 用語 | 意味 |
|---|---|
| ランナーオーナー（オーナー） | wasm_runner を自身のためにデプロイし、WASM の実行を行う主体。デプロイ時（初回時）に自身の Ed25519 公開鍵を1つ登録する（§7.1） |
| アップローダ（データ提供者） | データを登録する主体。任意の Ed25519 鍵で署名して登録し、その公開鍵で識別される。オーナーと同一人物でも別主体でもよい |
| ユーザ鍵 | 主体を識別する Ed25519 鍵ペア。事前登録は不要で、公開鍵の提示＋署名検証による自己認証的な識別を行う（§7.1）。恒久的な owner_id / API キーの発行は行わない |
| program_id | 事前アップロードされた WASM プログラムの識別子。`p-<sha256(バイナリ) hex>` のコンテンツアドレス（§7.2） |
| ホワイトリスト（allowed_programs） | データごとに保持する「このデータに対して実行を許可する program_id の集合」。編集はアップローダ専権（§7.2） |
| データID | 登録されたデータ1件を一意に識別する ID |
| TEE | Trusted Execution Environment。本プロジェクトでは Gramine-SGX によるエンクレーブ |
| 封印ストレージ | TEE 管理下で暗号化保存されるデータストア |
| 削除証明（Deletion Certificate） | データが削除されたことを TEE が署名して証明する電子的な証跡 |
| リモートアテステーション | SGX のクォートにより、エンクレーブの真正性・完全性を第三者が検証できる仕組み |
| クリプトシュレッディング | 暗号鍵を破棄することで、暗号化データを実質的に復元不能にする削除手法 |

## 3. スコープ

### 対象

- `wasm_runner`（Go / wazero / Gramine-SGX）へのライフサイクル管理機能・実行制約・
  削除証明機能の追加。

### 対象外（本書では扱わない）

- WASM プログラムのビルド環境（`wasm_module/`）自体の変更。
- 削除証明の検証者（third-party verifier）側の実装。本書は証明の**発行**のみを扱う。
- ユーザ鍵（Ed25519）の生成・保管・失効はユーザ側の責任とし、本書では扱わない。
  オーナー鍵のローテーション（TOFU 登録後の変更）も非対応（§11）。
- SGX の既知のサイドチャネル攻撃（キャッシュタイミング等）への対策。ハードウェア・
  Gramine 側の対策に委ね、本書のスコープ外とする。
- 出力内容の検査（マスキング／DLP 的な仕組み）。アップローダとオーナーが別主体に
  なり得る（2026-07-10 改訂）が、出力の保護は**データごとのホワイトリスト**（§7.2:
  アップローダが承認したコードだけが自分のデータを読める）で行い、出力そのものの
  検査は導入しない。検査では変換されたコピー（暗号化・符号化・要約等）を原理的に
  検出できず不完全な対策にしかならないため（§10・§11）。

### 信頼の前提（Trust Assumptions）

- Intel SGX のハードウェア保証（メモリ暗号化・アクセス制御）を信頼する。
- Gramine の libOS 実装を信頼する。
- `wasm_runner` バイナリ自体は計測・アテステーションの対象であり、改ざんされていない
  ことをリモートアテステーションで検証可能とする。
- インフラ管理者（root 含む）・ネットワーク経路上の攻撃者は**信頼しない**（脅威主体）。
- 実行できるのはランナーオーナーのみであり、その本人性は登録済みオーナー鍵による
  リクエスト署名で保証する（§7.1）。オーナー鍵の登録は TOFU（初回 `POST /owner`）で
  あり、デプロイから登録までの先取りリスクは「起動→即登録」の運用で扱う（§11）。
- アップローダはオーナーを無条件には信頼しない。アップローダが信頼するのは
  「自分が明示的に許可した（＝監査したコンテンツハッシュの）プログラムの出力が
  オーナーに渡ること」のみであり、これはデータごとのホワイトリスト（§7.2）で
  システム的に強制される。ただし**許可したプログラムの出力はオーナーに開示される**
  ことをアップローダは了解する（§10）。

## 4. 全体アーキテクチャ

```
                         ┌───────────────────────────────────────────────────┐
                         │        wasm_runner (Gramine-SGX enclave)          │
                         │                                                   │
  Client                 │  ┌─────────────────┐   ┌───────────────────────┐  │
 (data owner / user)     │  │ Lifecycle       │   │ Secure Storage        │  │
 ──── HTTP ────────────> │  │ Manager         │<─>│ (sealed data store,   │  │
   register/execute/     │  │ (state machine, │   │  e.g. Gramine         │  │
   delete                │  │  locking)       │   │  Protected Files)     │  │
                         │  └──────┬──────────┘   └───────────────────────┘  │
                         │         │                                         │
                         │         V                                         │
                         │  ┌─────────────────┐   ┌───────────────────────┐  │
                         │  │ WASM Execution  │   │ Attestation /         │  │
                         │  │ Sandbox         │   │ Deletion Proof        │  │
                         │  │ (wazero,        │   │ Module                │  │
                         │  │  network        │   │ (SGX quote            │  │
                         │  │  blocked / FS   │   │  generation,          │  │
                         │  │  restricted)    │   │  certificate signing) │  │
                         │  └─────────────────┘   └───────────────────────┘  │
                         └───────────────────────────────────────────────────┘
```

`wasm_runner` プロセス自体がエンクレーブ内で動作するため、ライフサイクル管理・
封印ストレージ・実行サンドボックス・証明発行のすべてが TEE の保護下にある。
外部との通信経路は HTTP（クライアントとのやり取り）のみであり、WASM 実行中は
ネットワークアクセスを提供しないことで、実行中のプログラムからの外部送信を
物理的に不可能にする（詳細は §8）。

## 5. データのライフサイクル状態遷移

```
                register
  (nonexistent) ────────> REGISTERED ──────────> IN_USE
                              ^                    │
                              └────────────────────┘
                              execute complete
                              │
                              │ delete
                              V
                          DELETING ─────────> DELETED (terminal)
                       (sealed data erasure + deletion certificate issuance)
```

| 状態 | 説明 | 許可される操作 |
|---|---|---|
| `REGISTERED` | データが封印保存され、利用可能な待機状態 | `execute`, `delete`, `status` |
| `IN_USE` | 特定の WASM プログラムを実行中（一時状態） | `status` のみ（`execute`/`delete` は排他） |
| `DELETING` | 削除処理中（封印データの消去・証明生成） | `status` のみ |
| `DELETED` | 削除完了。終端状態、データIDの再利用不可 | `status`, `proof`（削除証明の再取得） |

`execute` は1回の実行で複数のデータIDを同時に指定できる（0個以上の可変長、§7）。
この場合、指定された全データが原子的（all-or-nothing）に `IN_USE` へ遷移し、
実行完了時に全件が `REGISTERED` に戻る。1件でも取得できない場合は、どのデータの
状態も変更しない。

### 不変条件

1. 生値（平文）を直接返す専用 API（例: `GET /data/{id}`）は**一切存在しない**。
   `register` は書き込み専用、`execute` は WASM プログラムの計算結果のみを返す。
   ただし `execute` の結果内容自体は WASM プログラム次第であり、生データがそのまま
   出力される可能性を排除しない。実行できるプログラムをアップローダが明示的に
   許可したもの（ホワイトリスト §7.2）に限定することで、「アップローダが出力の
   開示を承認したコード」以外がデータに触れる経路を塞ぐ（§8-3, §10 参照）。
2. `IN_USE` 中は同一データIDに対する他の `execute`/`delete` をブロックする
   （データIDごとの排他ロック）。実行途中の状態に対する競合操作
   （TOCTOU: Time-of-check to time-of-use）を防ぐため。
3. `DELETED` は終端状態であり、以後そのデータIDへの `execute` は 404 を返す。
   同じ ID の再登録も禁止する（監査ログとの対応を一意に保つため）。
4. 状態遷移はすべて `wasm_runner` 内の単一の Lifecycle Manager が管理し、
   状態を書き換えられる経路を一本化する。

## 6. コンポーネント設計

### 6.1 Lifecycle Manager

- データIDごとに `{state, storage_ref, uploader, allowed_programs, content_hash, created_at}`
  を保持するメタデータストアと、状態機械の遷移ロジックを持つ
  （uploader はアップローダ公開鍵、allowed_programs はホワイトリスト。§7.2）。
- データIDごとの排他制御（mutex）により、`IN_USE`/`DELETING` 中の競合操作を防止する。
- メタデータ自体もプロセス再起動で失われてよいものではないため、封印ストレージ側に
  永続化する（§6.2）。

### 6.2 封印ストレージ（Secure Storage）

- 登録されたデータは平文のままディスクに置かず、暗号化して保存する。
- Gramine が提供する **Protected Files**（`sgx.protected_files`）機能の利用を第一候補
  とする。ファイル単位で AES-GCM 相当の暗号化・完全性検証が行われ、鍵はエンクレーブ
  外に露出しない。
- 鍵のプロビジョニング方式（固定鍵 / RA-TLS 経由の動的配布）は §11 の未解決課題とする。
- 削除は物理的なファイル消去に加え、対応する暗号鍵・メタデータを破棄する
  **クリプトシュレッディング**を正とする（§9）。

### 6.3 WASM 実行サンドボックス

- 現行の `handleExecuteWasm`（`wasm_runner/main.go`）をベースに、以下を追加する。
  - 実行対象データを読み取り専用のスコープ付きファイルとして WASM 側にマウント。
  - ネットワーク関連の WASI ホスト関数を一切 instantiate しない。
  - 実行タイムアウト・メモリ上限を設定する。
- 詳細は §8。

### 6.4 Attestation / Deletion Proof Module

- SGX のリモートアテステーション機構（`/dev/attestation/*` 経由の quote 生成）を
  用いて、削除イベントに対する証明を発行する。
- 詳細は §9。

## 7. API 設計（2026-07-10 改訂）

WASM 実行は単一の `POST /execute` に統合されている。実行するプログラムは事前に
`POST /programs` でアップロードし、execute では **program_id** で参照する。
**クエリパラメータは全 API で使用せず**、リクエストのパラメータはすべて JSON
ボディに含める（署名対象のボディハッシュで一括して改ざんから保護するため。§7.1）。

| メソッド / パス | 説明 |
|---|---|
| `GET /` | ヘルスチェック（認証不要） |
| `POST /owner` | オーナー公開鍵の登録（TOFU: 未登録時の初回のみ。登録済みは 409）。ボディ: `{ public_key }` |
| `POST /programs` | WASM プログラムの事前アップロード（オーナー署名必須）。ボディ: `{ wasm }`（base64）。応答: `{ program_id, uploaded_at }`。program_id は `p-<sha256>` で冪等 |
| `DELETE /programs/{id}` | プログラムの削除（オーナー署名必須） |
| `POST /execute` | WASM 実行（オーナー署名必須。ステートレス実行も含む）。ボディ: `{ program_id, data, args }`。応答: 実行結果 |
| `POST /data` | データ登録（任意の Ed25519 鍵の署名必須）。ボディ: `{ data, allowed_programs }`。応答: `{ data_id, registered_at }` |
| `PUT /data/{id}/programs` | ホワイトリストの全置換（記録済みアップローダ鍵の署名必須。オーナー鍵では不可） |
| `DELETE /data/{id}` | データ削除（オーナー鍵または記録済みアップローダ鍵の署名必須）。応答: 削除証明（JSON） |
| `GET /data/{id}/status` | 現在の状態を返す（認証不要）。生データは返さない |
| `GET /data/{id}/proof` | `DELETED` 状態のデータに対し、削除証明を再取得（監査用、認証不要） |

### 7.1 署名によるユーザ認証（API キー方式の廃止）

- 主体（オーナー・アップローダ）は **Ed25519 鍵ペア**で識別される。事前のユーザ
  発行（旧 `POST /users`）・API キーは**廃止**し、公開鍵の提示＋リクエスト署名の
  検証による自己認証的な識別を行う。恒久的な乱数 owner_id も発行しない。
- **オーナー鍵の登録（TOFU）**: デプロイ直後の初回 `POST /owner` でオーナー公開鍵を
  1つ登録し、以後変更不可。未登録の間は全コマンドを拒否する。初回登録リクエストの
  先取りリスクは「起動→即登録」の運用で扱う（§11）。
- **署名プロトコル**: コマンドは `X-Public-Key` / `X-Signature` / `X-Timestamp`
  ヘッダを付与し、`<METHOD>\n<PATH>\n<timestamp>\nsha256(body)` への署名を検証する。
  リプレイ対策は timestamp 許容ウィンドウ + 使用済み署名キャッシュ
  （TEE 内の時計がホスト由来である制約は受容。詳細は `wasm_runner/SPEC.md` §4.1）。
- **認可**: 認証（署名の正当性）と認可（その鍵に操作が許されるか）を分離する。
  execute・プログラム管理はオーナー鍵のみ。データ削除はオーナー鍵または当該データに
  記録済みのアップローダ鍵。ホワイトリスト編集は記録済みアップローダ鍵のみ。

`POST /execute` の詳細:

- データ指定0個はステートレス実行（動作確認用）であり、ライフサイクル管理・
  削除証明の対象にはならない（登録されていないデータに削除証明という概念は
  そもそも成立しないため）。ステートレス実行にもオーナー署名は必要。
- 複数データを指定した場合、指定した全データが実行中 `IN_USE` になる（§5）。
  取得は all-or-nothing とし、1件でも存在しない・削除済み・使用中・ホワイトリスト
  不一致の場合は、どのデータの状態も変更せずエラーを返す。同一IDの重複指定は
  拒否する。
- 指定順の i 番目のデータが読み取り専用ファイル `/data/input<i>` として WASM から
  見える（§8-2）。
- 認可は2段階: (1) リクエスト署名がランナーオーナーの鍵であること、
  (2) 指定した各データについて `program_id ∈ allowed_programs` であること（§7.2）。

### 7.2 プログラムレジストリとデータごとのホワイトリスト

- **program_id はコンテンツアドレス**（`p-<sha256(バイナリ) hex 小文字64桁>`）。
  同一バイナリの再アップロードは冪等に同じ ID を返す。ID がコード内容と1対1で
  結び付くため、アップローダはバイナリを事前に監査し、手元で計算したハッシュを
  ホワイトリストに載せることができる（ID の発行をサーバに信頼委譲しない）。
- データは登録時に `allowed_programs`（実行を許可する program_id の集合）を持つ。
  **未指定の既定は空リスト＝すべて拒否（deny by default）**。
- ホワイトリストの編集（`PUT /data/{id}/programs`、全置換）は**アップローダ専権**。
  オーナー鍵では編集できない（オーナーが自分に都合よく許可を広げることを防ぐ）。
- ハッシュ参照のため、その時点で未アップロードのプログラムもホワイトリストに
  載せられる（アップロード順序に制約はない）。
- アテステーション（ランナー自体の検証）と組み合わせると、アップローダは
  「自分が監査・承認したコードだけが自分のデータを読める」ことをエンドツーエンドで
  保証できる。残る了解事項: **許可したプログラムの出力はオーナーに渡る**（§10）。
- プログラムはデータと異なりライフサイクル管理・削除証明の対象ではない。

## 8. WASM プログラムの実行制約

「任意のプログラムを実行できるが、外部送信はできない」を実現するための制約。

1. **ネットワークアクセスの禁止**
   - wazero に WASI Socket 拡張やその他ネットワーク系ホスト関数を一切登録しない。
     現行実装は `wasi_snapshot_preview1` の Core Module 相当のみを instantiate して
     おり、素の状態でもソケット API は提供されていない。これを維持し、将来的にも
     ネットワーク系ホスト関数を追加しないことを設計原則として明文化する。
   - 多層防御として、Gramine のマニフェスト側でも該当プロセスからのネットワーク
     syscall 発行経路を最小化する設定を検討する。

2. **ファイルシステムアクセスの制限**
   - `wazero.NewModuleConfig().WithFS(...)` 等を用い、当該実行で指定された登録
     データ（0個以上、§7）のみを、指定順に読み取り専用で `/data/input0`,
     `/data/input1`, ... としてマウントする。データ指定なしの実行では FS 自体を
     与えない。
   - ホストの他ディレクトリへの経路は一切与えない。書き込み用マウントも提供しない
     （結果は stdout 経由でのみ返す）。

3. **出力サイズの制限**
   - 既存の `cappedBuffer`（`outputCap = 1 MiB`）を維持する。ただしこれはリソース
     枯渇（DoS）防止のための制限であり、機密性の担保を目的としたものではない。
   - 実行結果（stdout）はランナーオーナーに返る。データに触れられるのは
     アップローダがホワイトリストで明示的に許可したプログラムのみ（§7.2）であり、
     「許可したプログラムの出力はオーナーに開示される」ことはアップローダの
     了解事項とする（§10）。したがって出力内容を検査するポリシーチェック
     （DLP 的な仕組み）は導入しない。変換されたコピー（暗号化・符号化・要約等）は
     原理的に検出できず不完全な対策にしかならないため（§11）。

4. **リソース制限**
   - 現行実装には実行タイムアウトが存在しないため、`context.WithTimeout` を
     `InstantiateModule` に適用する。
   - `wazero.NewRuntimeConfig().WithMemoryLimitPages(...)` 等でメモリ上限を設定し、
     無限ループやメモリ枯渇によるサイドチャネル・DoS を緩和する。

5. **最小権限の原則**
   - instantiate する WASI ホスト関数は必要最小限に絞る（`args`, `environ`,
     `clock`, `random` 等は用途を精査し、不要であれば無効化する）。

## 9. 削除証明（Deletion Proof）の設計

### 9.1 削除の実体

- 封印ストレージ上の暗号化データを消去するとともに、対応する暗号鍵・メタデータを
  破棄する（クリプトシュレッディング）。暗号化 blob 自体がストレージ上に残存しても、
  復号不能であれば実質的に削除されたとみなす。

### 9.2 削除証明のフォーマット（例）

```json
{
  "data_id": "d-xxxxxxxx",
  "deleted_at": "2026-07-08T12:34:56Z",
  "content_hash": "sha256:...(登録時点で記録した元データのハッシュ)",
  "enclave_report": {
    "mrenclave": "...",
    "mrsigner": "...",
    "quote": "base64:..."
  },
  "signature": "base64:...(エンクレーブが保持する鍵による署名)"
}
```

### 9.3 検証手順（第三者検証者側、参考）

1. `quote` を Intel PCS/DCAP 等で検証し、`mrenclave`/`mrsigner` が期待値と一致する
   ことを確認する。
2. エンクレーブ起動時に一度だけ発行された quote の `user_report_data` に、以後の
   署名に使う公開鍵のハッシュが埋め込まれていることを確認する（毎回 quote を
   生成するコストを避けるため、起動時に一度だけ鍵を証明し、以後はその鍵で署名する
   運用とする）。
3. `signature` を当該公開鍵で検証する。
4. `content_hash` が登録時に取得したハッシュと一致することを確認し、証明対象の
   データが登録されたデータそのものであることを確認する。
5. 以上がすべて成立すれば、「当該データはこの（真正な）エンクレーブ内で削除された」
   ことが第三者に対して証明される。

## 10. 脅威モデルと対策

| 脅威 | 対策 |
|---|---|
| ホスト OS / インフラ管理者によるメモリ盗聴・改ざん | SGX のメモリ暗号化・アクセス制御 |
| 悪意ある WASM プログラムによるネットワーク経由のデータ持ち出し | ネットワーク系ホスト関数を一切提供しない（§8-1） |
| オーナーがアップローダの承認していないプログラムでデータを読む（stdout 経由の持ち出しを含む） | データごとのホワイトリスト（§7.2）: `program_id ∈ allowed_programs` でないデータ指定の execute は 403。program_id はコンテンツハッシュなので、許可＝特定のコード内容の承認 |
| アップローダが許可したプログラムの出力がオーナーに渡る | 脅威ではなく了解事項: アップローダは「出力が開示されてよいプログラムだけを許可する」責任を負う（生データをそのまま出力するプログラムを許可すれば、実質的に生データをオーナーに開示したことになる） |
| 第三者によるコマンドの偽造・改ざん・リプレイ | Ed25519 リクエスト署名（メソッド・パス・タイムスタンプ・ボディハッシュに署名）+ timestamp ウィンドウ + 使用済み署名キャッシュ（§7.1）。秘密鍵はサーバに送られないため、サーバ側からの鍵漏洩は構造上ない |
| オーナー登録（TOFU）の先取り | 運用対策のみ: デプロイ直後に即座に登録する。先取りされた場合はコマンドが通らなくなることで即検知でき、データ登録前なら `data_store/` を消して再デプロイすればよい（§11） |
| ホワイトリストの不正な書き換え（オーナーによる自己認可） | 編集はデータに記録済みのアップローダ鍵の署名のみ受理（§7.2）。オーナー鍵では編集不可 |
| 削除証明の偽造 | SGX リモートアテステーションによる署名鍵の真正性保証（§9） |
| `IN_USE`/`DELETING` 中の競合操作（TOCTOU） | データIDごとの排他ロックと厳格な状態機械（§5） |
| 削除済みデータのバックアップ等からの復元 | クリプトシュレッディングにより鍵を破棄し、平文的価値を除去（§9.1） |
| SGX サイドチャネル攻撃 | 本書のスコープ外（ハードウェア/Gramine 側の対策に委ねる） |

## 11. 制限事項・未解決課題

- **利用者≠データオーナーの信頼モデル**: 2026-07-10 改訂でアップローダと
  ランナーオーナーは別主体になり得る。stdout 経由の情報持ち出しは「実行可能な
  処理を事前承認されたコード（コンテンツハッシュ）に限定する」ホワイトリスト
  （§7.2）で扱う——これは当初 §11 が予告していた「別の保護機構」の実装に当たる。
  残る限界: 許可したプログラムの出力はオーナーに開示される（アップローダの許可
  判断の質に依存する）。差分プライバシー的な出力制約などの上乗せは未対応。
- **第三者検証者の実装**: 本書はエンクレーブ側での証明**生成**のみを扱う。検証者側
  ツールの実装は別プロジェクトとする。
- **TOFU の先取りリスク**: `POST /owner` は認証不要のため、デプロイから登録までの
  間に攻撃者が先に鍵を登録できる。対策は運用（起動→即登録の一連化）のみ。
  案A（マニフェストの環境変数でオーナー鍵を与え、計測に含める）なら排除できるが、
  鍵変更に再デプロイが必要になるため現時点では不採用（計画書 §7-Q1 で受容）。
- **時計の信頼とリプレイ**: 署名タイムスタンプの検証に使う時計はホスト由来であり、
  ホストが時計を操作すればリプレイウィンドウを広げられる。また使用済み署名
  キャッシュはメモリ上のみで、プロセス再起動を跨ぐとウィンドウ幅ぶんの再利用余地が
  ある。脅威評価に応じてサーバ発行 nonce（チャレンジ・レスポンス）方式への移行を
  検討する（計画書 §7-Q2）。
- **鍵のローテーション**: オーナー鍵（TOFU 登録後）・アップローダ鍵（データに
  記録後）の変更・失効 API は未実装。
- **鍵プロビジョニング**: Gramine Protected Files の暗号鍵をどう配布するか
  （固定鍵 / RA-TLS 経由の動的配布）は要検討。`owner.json` を含むメタデータの
  ロールバック（ホストによるファイル巻き戻し）も従来どおり未対策。

## 12. 実装計画（マイルストーン案）

1. **Phase 1**: ライフサイクル状態機械 + `register`/`execute`/`delete`/`status` API
   の実装（暗号化なし、まずは状態遷移とロックの正しさを確認）。
2. **Phase 2**: Gramine Protected Files による封印ストレージの導入。
3. **Phase 3**: wazero 実行制約の強化（FS スコープ限定、タイムアウト、メモリ上限）。
4. **Phase 4**: SGX リモートアテステーションによる削除証明の署名・発行機能。

## 13. 現行実装との差分まとめ

- **現状**（`wasm_runner/main.go`）: `POST /execute-wasm` はステートレスであり、
  送信された WASM バイナリをその場で wazero 上で実行し、stdout/stderr を返すのみ。
  データの永続化・ライフサイクルという概念自体が存在しない。
- **拡張後**: データの登録・利用・削除を Lifecycle Manager が状態機械として管理し、
  WASM 実行は wazero の機能（`WithFS`、メモリ上限、タイムアウト等）とネットワーク
  ホスト関数の非提供により厳格に制約される。削除時には SGX リモートアテステーション
  に基づく削除証明が発行される。WASM 実行 API は `POST /execute` に一本化され、
  1回の実行で使用する登録済みデータを可変長（0個以上）で指定できる（§7）。
- **2026-07-10 改訂後**: 認証は Ed25519 リクエスト署名（API キー廃止）。WASM は
  `POST /programs` による事前アップロード（`p-<sha256>` のコンテンツアドレス）に
  変わり、execute は program_id 参照。データはアップローダの公開鍵で識別され、
  データごとの実行可能プログラムホワイトリスト（deny by default・アップローダ専権
  編集）で「誰のどのコードが実行できるか」を制御する。as-built の詳細は
  `wasm_runner/SPEC.md` を参照。
