# wasm_runner アップデート作業整理（Ed25519 署名認証・WASM 事前アップロード）

作成日: 2026-07-10
関連文書: `docs/260709-data-lifecycle-spec.md`（設計書）, `wasm_runner/SPEC.md`（as-built 実装仕様）

## 0. 本書の位置づけ

wasm_runner に対する次の3つのアップデート要求を、現行実装との差分・設計論点・
実装タスクに分解して整理したものである。**実装前の作業計画書**である。
当初の要決定事項（§7）は 2026-07-10 にすべて確定した（§7 に決定を記す）。

## 1. アップデート要求（原文の整理）

| # | 要求 | 内容 |
|---|---|---|
| U1 | wasm_runner のデプロイ | ユーザは自身が利用する wasm_runner をデプロイする時（初回時）に、自身の公開鍵を登録する。WASM プログラムを実行するとき（コマンドを送信するとき）はユーザ認証を必須とする。認証は登録済みの鍵でコマンドに対する署名を作ることで行う |
| U2 | WASM プログラムの事前アップロード | 従来は execute 時に WASM バイナリを送信していたが、事前にアップロードできるようにする。アップロード時に ID を発行し、execute 時は ID を指定して実行する |
| U3 | データアップロード時の署名 | データのアップロードは従来どおりだが、署名を必須とする。この鍵はデプロイ時の鍵（U1）と同一でなくてよい。誰がデータをアップロードしたかの**識別**に利用する |
| U4 | 鍵種 | ユーザを識別する鍵は Ed25519 を利用する |
| U5 | データごとの実行可能プログラム指定（ホワイトリスト） | アップローダは、アップロードしたデータをどの WASM プログラムで実行できるかを指定できる。データごとに実行可能な program_id のホワイトリストを持ち、**編集はアップローダのみ**可能。program_id は WASM バイナリのハッシュ値を用いた **`p-<hash>` 形式**とする（2026-07-10 追加。§3.3「設計上の重要な影響」への対応方針として決定） |

## 2. 現行実装の要点（変更対象の把握）

| 項目 | 現状（`wasm_runner/`） |
|---|---|
| 認証 | `POST /users` が owner_id + API キーを発行。sha256(キー) を `users.json` に保存し、`X-API-Key` / `Authorization: Bearer` ヘッダで認証（`users.go`） |
| 認可 | データレコードの `owner_id` と認証済みユーザを照合（execute/delete）。前提は「利用者＝データオーナー本人」 |
| execute | `POST /execute` の JSON ボディに WASM バイナリを base64 で毎回送信（`wasm` フィールド、最大 32 MiB → ボディ上限 44 MiB） |
| データ登録 | `POST /data`（認証必須）。ボディ＝生データ。登録者の owner_id をメタデータに記録 |
| 鍵 | ユーザ側に鍵ペアは存在しない（API キーのみ）。Ed25519 はサーバ側の削除証明署名（`attestation.go`）で既に使用 |

## 3. 変更内容の整理

### 3.1 U1: オーナー公開鍵の登録と署名によるコマンド認証

**変更後の姿**

- wasm_runner は「1ユーザが自身のためにデプロイするサーバ」となり、デプロイ時
  （初回時）に**オーナーの Ed25519 公開鍵を1つ登録**する。
- コマンド（execute を含む、状態を変える操作）はすべて、登録済みオーナー鍵による
  **リクエストへの署名**で認証する。API キー方式（`POST /users` / `users.json` /
  `X-API-Key`）は**廃止**する。
- オーナーの識別子は公開鍵そのもの（または `sha256(公開鍵)` の指紋）とし、
  乱数 owner_id（`u-...`）の発行は不要になる。

**鍵登録の方式（決定: 案B → §7-Q1）**

| 案 | 方法 | 利点 | 欠点 |
|---|---|---|---|
| A | 起動時に環境変数 `OWNER_PUBKEY`（base64）で与える。Gramine マニフェストの `loader.env` に記載 | マニフェスト計測に公開鍵が含まれ、アテステーションで「この鍵のオーナーのランナーである」ことまで検証可能。登録 API が不要で「初回時のみ」の制御が自明 | 鍵変更にはマニフェスト再生成・再デプロイが必要 |
| **B（採用）** | 初回起動後、最初の `POST /owner`（TOFU: Trust On First Use）で登録し、以後は受け付けない | 再デプロイ不要で鍵を投入できる | 初回登録リクエストを攻撃者に先取りされるリスク。登録済み状態の永続化と「初回」判定が必要 |

案B の実装方針:

- `POST /owner`: ボディ＝JSON `{"public_key": "<base64>"}`。未登録のときのみ受け付け、
  公開鍵を検証（base64・raw 32B）して封印ストレージに永続化（例: `owner.json`。
  原子的書き込みは meta と同じ方式）。登録済みなら **409** で拒否し、以後変更不可。
- 登録前は `POST /owner` と認証不要 API（`GET /` 等）以外のコマンドをすべて拒否する
  （未登録状態でコマンドが通る窓を作らない）。
- 先取りリスクへの運用上の対策: デプロイ直後に即座に登録する（デプロイ手順に
  組み込む。T10 のスクリプトで起動→登録を一連で行う）。
- 案A と異なりオーナー鍵はマニフェスト計測に含まれないため、アテステーションで
  検証できるのはランナーのコードまでとなる点は許容する（受容済みのトレードオフ）。

**署名プロトコル（決定: リプレイ対策は (a) timestamp 方式 → §7-Q2）**

リクエストに次のヘッダを付与する:

```
X-Public-Key: <Ed25519公開鍵 raw 32B の base64>
X-Signature:  <署名の base64>
X-Timestamp:  <RFC3339 UTC>（必須）
```

署名対象（メッセージ）は決定的に再構築できる形に正規化する:

```
<METHOD>\n<PATH>\n<X-Timestamp>\nsha256(<request body>)
```

- **クエリパラメータは全 API で使用しない（決定）**。リクエストのパラメータは
  すべて JSON ボディに含める。したがって `<PATH>` はパスのみで、パラメータは
  ボディハッシュを通じて署名で改ざんから保護される（クエリの正規化問題も
  発生しない）。
- ボディ全体でなくボディのハッシュに署名することで、大きなボディ（最大約 44 MiB、
  §3.2）でも署名検証コストを一定にする。
- リプレイ対策は **(a) timestamp + 許容ウィンドウ + 使用済み署名キャッシュ**を
  採用する。TEE 内の時計はホスト由来で信頼できない点は既知の制約として受容し、
  脅威評価に応じて将来 (b) サーバ発行 nonce（チャレンジ・レスポンス）への移行を
  検討する。
- 検証は Go 標準の `crypto/ed25519` で行う（追加依存なし）。

**認証の適用範囲（決定済み）**

| API | 認証 |
|---|---|
| `POST /owner`（新設） | 認証不要。ただし**未登録時の初回のみ**受け付ける（TOFU。登録済みは 409） |
| `POST /execute` | **オーナー鍵の署名必須**。ステートレス実行（data 指定なし）も含めて統一（§7-Q4）。実行できるのはオーナーのみ |
| `POST /programs`（新設、§3.2） | オーナー鍵の署名必須（コマンドの一種） |
| `DELETE /programs/{id}`（新設、§3.2） | **オーナー鍵の署名必須**（プログラムを削除できるのはオーナーのみ。§7-Q5） |
| `DELETE /data/{id}` | **オーナー鍵または当該データに記録済みのアップローダ鍵**の署名必須（§7-Q3: 両方可） |
| `POST /data` | **任意の Ed25519 鍵**の署名必須（§3.3。オーナー鍵である必要はない） |
| `PUT /data/{id}/programs`（新設、§3.4） | **当該データに記録済みのアップローダ鍵**の署名必須。オーナー鍵では不可 |
| `GET /`・`status`・`proof` | 認証不要（現行どおり。生データ・鍵を含まないため） |

### 3.2 U2: WASM プログラムの事前アップロード（プログラムレジストリ）

**変更後の姿**

- 新設 `POST /programs`: ボディ＝JSON `{"wasm": "<base64>"}`（パラメータをすべて
  JSON ボディに含める全体方針に従う。§3.1）。応答で **program_id** を返す。
  オーナー鍵の署名必須。
- program_id は **`p-<sha256(バイナリ) の hex 小文字64桁>`** とする
  （コンテンツアドレス方式。U5 の要求）。乱数 ID は発行しない。
  - 同一バイナリの再アップロードは同じ ID になる（**冪等**。既存なら上書きせず
    同じ ID を 200 で返す）。衝突時の再生成ロジックは不要。
  - ID がコード内容と1対1で結び付くため、アップローダは**バイナリを事前に監査し、
    手元で計算したハッシュをホワイトリスト（§3.4）に載せる**ことができる。
    ID の発行をサーバに信頼委譲する必要がない。
  - ロード時に blob の sha256 を ID と突き合わせて整合性を検証できる
    （encrypted mount の保護に加えた二重チェック）。
- `POST /execute` の JSON ボディは `wasm`（base64）に代えて `program_id` を指定する:

  ```json
  { "program_id": "p-...", "data": ["d-...", "..."], "args": ["..."] }
  ```

- 副次効果: execute のボディから base64 の WASM が消え、execute ボディは小さな
  JSON のみになる。一方、全 API を JSON ボディに統一したことで base64 による
  約 4/3 倍の膨張は `POST /programs` と `POST /data` 側に移る（それぞれボディ上限
  約 44 MiB。現行 `maxExecBodyBytes` と同水準であり、エンクレーブサイズ上の
  新たな懸念は生じない）。

**プログラムの管理方針**

- 保存先は封印ストレージ（現行 `store` を流用。例: `programs/<id>.bin` +
  `programs/<id>.json` のメタデータ）。Gramine encrypted mount の保護下に入る。
- メタデータ: `program_id`（＝ `p-` + sha256）/ `size` / `uploaded_at` /
  アップロード者（＝オーナー指紋）。sha256 は ID に含まれるため独立フィールドは不要。
- データ（`/data`）のような状態機械・削除証明は**設けない**（プログラムは
  ライフサイクル管理・削除証明の対象ではないため）。put / 参照 / 削除のみとする。
- `DELETE /programs/{id}` を設ける。**削除できるのはオーナーのみ**（オーナー鍵の
  署名必須。§7-Q5）。一覧 API（`GET /programs`）は当面設けない（必要になれば追加）。
  コンテンツアドレス方式のため、削除後に同一バイナリを再アップロードすれば同じ
  ID に復元される（ホワイトリスト側の参照は書き換え不要）。
- 上限は WASM バイナリ実体で現行 `maxWasmBytes`（32 MiB）を踏襲
  （base64 込みのボディ上限は約 44 MiB）。

### 3.3 U3: データアップロード時の署名（アップローダ識別）

**変更後の姿**

- `POST /data` はボディを JSON `{"data": "<base64>", "allowed_programs": ["p-...", ...]}`
  に変更し（パラメータをすべて JSON ボディに含める全体方針。§3.1）、
  **任意の Ed25519 鍵による署名を必須**とする（署名プロトコルは §3.1 と共通）。
  事前のユーザ登録は不要で、公開鍵の提示＋署名検証のみ（自己認証的な識別）。
- データのメタデータ（`metaRecord`）に、現行の `owner_id` に代えて
  **アップローダの公開鍵（または指紋）**を記録する。目的は「誰がアップロード
  したかの識別」に加え、U5 により**ホワイトリスト編集の認可主体の特定**
  （§3.4。記録済みアップローダ鍵による署名のみ編集を受け付ける）。
  つまりアップローダ鍵の役割は識別のみでなく認可にも拡大する。
- 削除証明にアップローダ識別子は**含めない**（決定。§7-Q6。公開鍵が
  プライバシー上の準識別子になることを避ける。削除証明の内容は現行どおり）。

**設計上の重要な影響（既存前提の変更）→ U5 ホワイトリストで対応（決定）**

現行設計書（`docs/260709-data-lifecycle-spec.md` §3）は「利用者＝データオーナー
本人」を前提に、stdout 経由の生データ出力を許容していた。本アップデートで
**アップローダ（データ提供者）とランナーオーナー（実行者）が別主体になり得る**ため、
この前提は崩れる（オーナーは、他者がアップロードしたデータに対して WASM を実行し、
その出力＝生データを含み得るものを受け取れるようになる）。

これに対し「アップローダはオーナーがデータを読み得ることを了解する」という
信頼モデルへの単純変更は**採らず**、**データごとの実行可能プログラムの
ホワイトリスト（U5、§3.4）で対応する**:

- オーナーがデータに対して実行できるのは、**アップローダが明示的に許可した
  プログラムのみ**。ホワイトリスト外の program_id を指定した execute は拒否する。
- program_id はバイナリの sha256（§3.2）なので、許可は「特定のコード内容の承認」
  を意味する。アテステーション（ランナー自体の検証）と組み合わせると、
  アップローダは「自分が監査・承認したコードだけが自分のデータを読める」ことを
  エンドツーエンドで保証できる。
- 残る信頼モデル上の了解事項: **許可したプログラムの出力はオーナーに渡る**。
  アップローダは「出力が開示されてよいプログラムだけを許可する」責任を負う
  （例: 生データをそのまま出力するプログラムを許可すれば、実質的に生データを
  オーナーに開示したことになる）。
- execute 時の所有者照合（現行 `checkOwner`）は「データの owner_id ＝実行者」から
  「**実行者＝ランナーオーナー（署名）＋ データごとのホワイトリスト照合**」の
  2 段階に再定義される（T7）。データ単位の照合は不要になるのではなく、
  照合対象が「誰が」から「どのプログラムで」に変わる。
  設計書 §3・§7.1・§10 の該当箇所の改訂が必要。

### 3.4 U5: データごとの実行可能プログラムホワイトリスト

**決定事項（要求そのもの）**: データごとに実行可能な program_id のホワイトリストを
持つ。編集はアップローダのみ。program_id は `p-<sha256(バイナリ) hex>`（§3.2）。

**データモデル**

- `metaRecord` に `allowed_programs: []string`（program_id の集合）を追加する。
  ホワイトリストは状態機械の状態ではなく**可変メタデータ**であり、編集は
  状態遷移を伴わない（既存の原子的メタ書き込みで永続化）。
- ホワイトリストはコンテンツアドレス（ハッシュ）参照のため、**その時点で
  未アップロードのプログラムも載せられる**。アップロード順序に制約はない
  （execute 時に program_id が未登録なら通常どおり 404）。

**アップロード時の指定**

- `POST /data` の JSON ボディで指定する（クエリパラメータは使用しない。§3.1）:

  ```json
  { "data": "<base64>", "allowed_programs": ["p-<hash1>", "p-<hash2>"] }
  ```

  ホワイトリストはボディの一部なので、ボディハッシュを通じて署名で改ざんから
  保護される。
- 未指定（フィールド省略・空配列）の既定は**空リスト＝すべて拒否
  （deny by default）**とする（決定。§7-Q8）。アップローダが後から編集 API で
  許可を追加する運用。

**編集 API（新設）**

- `PUT /data/{id}/programs`: ボディ＝JSON `{"allowed_programs": ["p-...", ...]}`。
  **全置換**の意味論とする（追加・削除の差分 API は設けない。冪等で単純）。
  空配列で「すべて拒否」に戻せる。
- 認証: **データに記録済みのアップローダ公開鍵による署名のみ**受け付ける
  （T1 の第3の検証モード）。**オーナー鍵の署名では編集できない**（要求どおり
  アップローダ専権。オーナーが自分に都合よく許可を広げることを防ぐのが目的の
  ため、ここを許すと U5 が無意味になる）。
- 編集はレコードが削除系状態（deleting / deleted）に入るまで可能。実行中の編集は
  レコードロック下で直列化し、実行への適用タイミングは「execute 開始時点の
  スナップショット」とする（実行途中の変更は当該実行に影響しない）。

**execute 時の照合（T7 の再定義）**

1. リクエスト署名がランナーオーナーの鍵であること（U1）。
2. 指定された各データについて `program_id ∈ allowed_programs` であること。
   1 件でも外れていれば **403**（応答にどのデータが拒否したかを含める）。
3. データ指定なしのステートレス実行にはホワイトリストは関与しない
   （オーナー署名は必要。§7-Q4）。

**削除証明との関係**: ホワイトリストは削除証明の対象外（データ本体の
ライフサイクルにのみ証明を発行する現行方針を維持）。

### 3.5 U4: 鍵種

- ユーザ識別鍵は Ed25519（raw 32B 公開鍵、base64 表現）。
- サーバ側の削除証明署名も既に Ed25519 であり、鍵の表現・署名検証コードは共通化できる。

## 5. 実装タスク分解

依存順に並べる。T1 が全体の土台になる。

## 6. 互換性・移行

- **既存 `data_store/` とは非互換**: `users.json` は不要になり、`metaRecord.owner_id`
  の意味（乱数 ID → 公開鍵指紋）が変わり、`allowed_programs` フィールドが加わる。前回の認証方式変更時（SPEC.md §10）と
  同様、`data_store/` を空にして登録し直す方針とする（移行ツールは作らない）。
- 旧 API（`POST /users`、`X-API-Key`、execute の `wasm` フィールド）は 404 / 400 と
  なる（併存期間なしの一括置き換え。§7-Q7）。`POST /data` のボディも生データから
  JSON（base64）に変わるため、クライアント（scripts）の同時更新が必要（T10）。

## 7. 決定事項（2026-07-10 確定）

当初の要決定事項は以下のとおり確定した。あわせて「クエリパラメータは全 API で
使用せず、パラメータはすべて JSON ボディに含める」ことを決定した（§3.1）。

| # | 論点 | 決定 | 補足 |
|---|---|---|---|
| Q1 | オーナー鍵の登録経路 | **案B: 初回 TOFU API（`POST /owner`）** | 再デプロイ不要で鍵を投入できる。先取りリスクは「デプロイ直後に即登録」の運用で対処（§3.1）。オーナー鍵がマニフェスト計測に含まれない点は受容 |
| Q2 | リプレイ対策 | **(a) timestamp + 許容ウィンドウ + 使用済み署名キャッシュ** | TEE 内の時計がホスト由来である制約は受容。脅威評価に応じて将来 (b) nonce 方式を検討 |
| Q3 | `DELETE /data/{id}` の認可主体 | **(c) オーナーもアップローダも削除可能** | オーナー鍵または当該データに記録済みのアップローダ鍵の署名を受け付ける |
| Q4 | ステートレス実行（data 指定なし）の扱い | **オーナーのみ実行可能（署名必須に統一）** | execute は形態によらずオーナー署名必須。ホワイトリストは関与しない（§3.4） |
| Q5 | プログラムの削除・一覧 API | **`DELETE /programs/{id}` を設け、削除できるのはオーナーのみ** | 一覧 API は当面設けない。コンテンツアドレスのため再アップロードで同一 ID に復元可能（§3.2） |
| Q6 | 削除証明へのアップローダ識別子の記載 | **記載しない** | 公開鍵がプライバシー上の準識別子になることを避ける。削除証明の内容は現行どおり |
| Q7 | 旧 API キー方式の撤去 | **完全撤去（併存期間なしの一括置き換え）** | `POST /users` / `users.json` / `X-API-Key` を廃止（§6） |
| Q8 | ホワイトリスト未指定時の既定 | **空＝すべて拒否（deny by default）** | アップローダが `PUT /data/{id}/programs` で後から許可を追加する運用（§3.4） |

## 8. 実装マイルストーン案

1. **Phase 1（認証基盤）**: T1 + T2 + T9（認証系テスト）。既存 API を署名認証に
   置き換えるところまで（execute はまだ `wasm` 直接送信のまま動く状態を維持）。
2. **Phase 2（プログラムレジストリ）**: T3–T5。`POST /programs` と execute の
   ID 参照化。
3. **Phase 3（データ署名・認可再定義・ホワイトリスト）**: T6 + T7 + T7b。
   アップローダ識別、データごとのホワイトリスト（U5）、削除証明の扱い。
4. **Phase 4（統合・整備）**: T8 + T10 + T11。Gramine 実機確認（direct / SGX）と
   ドキュメント改訂。
