# データライフサイクル管理機能 拡張仕様書（wasm_runner）

## 0. 文書の位置づけ

本書は `wasm_runner`（TEE = Gramine-SGX 上で動作する WASM 実行サーバー）に、
データの**登録・利用・削除のライフサイクル管理**機能を追加するための設計仕様書である。
現行実装（`wasm_runner/main.go`）は「受け取った WASM バイナリをその場で実行して
結果を返すだけ」のステートレスなサーバーであり、データという概念自体を持たない。
本拡張により、`wasm_runner` は次の2つの役割を明確に分離して担う。

| 役割 | 担当 |
|---|---|
| データのライフサイクル管理（登録・利用・削除、削除証明の発行） | `wasm_runner`（ホスト側 Go プログラム） |
| ライフサイクル内で行う任意の計算処理 | 利用者が提出する WASM プログラム |

最終目的は、「登録されたデータが、いつ・誰によっても複製されず、最終的に削除された」
ことを第三者が検証可能な形で証明できるようにすること（**削除証明**）。

## 1. 背景・目的

- データの削除証明を成立させるには、データがシステム内のどこに存在し、どんな処理を
  経て、いつ消えたかという**ライフサイクル全体**を管理下に置く必要がある。
- ライフサイクルの管理（状態遷移・アクセス制御・削除の実施・証明の発行）は
  `wasm_runner` が TEE 内で行う。TEE を信頼の起点とすることで、インフラ管理者を
  含む外部者による干渉・盗聴からライフサイクル管理そのものを保護する。
- ライフサイクル内で実行される処理（データの集計・変換・学習など）は、利用者が
  自由に持ち込む WASM プログラムに任せる。ただし、その WASM プログラムが
  **データを外部に送信できてしまうと、複製が生成され削除証明が意味を失う**ため、
  外部送信を強制的に禁止する必要がある。
- 本書はこの要件を満たすための `wasm_runner` の拡張仕様を定義する。

## 2. 用語

| 用語 | 意味 |
|---|---|
| データオーナー | データを登録する主体 |
| 利用者 | 登録済みデータに対して WASM プログラムの実行を依頼する主体。本書では**データオーナー本人であることを前提とする**（別主体への実行委譲は非対応。§3・§10参照） |
| owner_id | ユーザ（データオーナー）を一意に識別する恒久的な識別子。秘密ではなく、データレコード・ログ・監査記録に載せてよい（§7.1） |
| API キー | ユーザが「自分が owner_id 本人である」ことを証明するための秘密。サーバはハッシュのみを保存し、発行時以外に平文が現れることはない（§7.1） |
| データID | 登録されたデータ1件を一意に識別する ID |
| TEE | Trusted Execution Environment。本プロジェクトでは Gramine-SGX によるエンクレーブ |
| 封印ストレージ | TEE 管理下で暗号化保存されるデータストア |
| 削除証明（Deletion Certificate） | データが削除されたことを TEE が署名して証明する電子的な証跡 |
| リモートアテステーション | SGX のクォートにより、エンクレーブの真正性・完全性を第三者が検証できる仕組み |
| クリプトシュレッディング | 暗号鍵を破棄することで、暗号化データを実質的に復元不能にする削除手法 |

## 3. スコープ

### 対象

- `wasm_runner`（Go / wazero / Gramine-SGX）へのライフサイクル管理機能・実行制約・
  削除証明機能の追加。

### 対象外（本書では扱わない）

- WASM プログラムのビルド環境（`wasm_module/`）自体の変更。
- 削除証明の検証者（third-party verifier）側の実装。本書は証明の**発行**のみを扱う。
- ユーザ発行（`POST /users`）自体の保護。認証は API キー方式（§7.1）を採用するが、
  「誰がユーザを発行できるか」の制限、および API キーのローテーション・失効の
  設計は別途とする（§11）。
- SGX の既知のサイドチャネル攻撃（キャッシュタイミング等）への対策。ハードウェア・
  Gramine 側の対策に委ね、本書のスコープ外とする。
- データオーナーと異なる主体への実行委譲（データマーケットプレイス的な用途）。本書は
  利用者＝データオーナー本人であることを前提とし、この前提のもとでは stdout 経由の
  出力は常にデータ保有者本人に返るため「意図しない第三者への漏洩」には当たらないと
  判断する。異なる主体への委譲を将来サポートする場合、出力内容の検査（マスキング／
  DLP的な仕組み）は変換されたコピー（暗号化・符号化・要約等）を原理的に検出できず
  不完全な対策にしかならないため、別の保護機構の再設計が必要になる（§11）。

### 信頼の前提（Trust Assumptions）

- Intel SGX のハードウェア保証（メモリ暗号化・アクセス制御）を信頼する。
- Gramine の libOS 実装を信頼する。
- `wasm_runner` バイナリ自体は計測・アテステーションの対象であり、改ざんされていない
  ことをリモートアテステーションで検証可能とする。
- インフラ管理者（root 含む）・ネットワーク経路上の攻撃者は**信頼しない**（脅威主体）。
- 利用者はデータオーナー本人であることを前提とする。この対応関係は、ユーザ認証
  （API キー → owner_id の解決）とデータレコードの owner_id 照合によって保証する
  （§7.1）。ユーザ発行の主体の制限は §3 の対象外に記載の通り別途とする。

## 4. 全体アーキテクチャ

```
                         ┌───────────────────────────────────────────────────┐
                         │        wasm_runner (Gramine-SGX enclave)          │
                         │                                                   │
  Client                 │  ┌─────────────────┐   ┌───────────────────────┐  │
 (data owner / user)     │  │ Lifecycle       │   │ Secure Storage        │  │
 ──── HTTP ────────────> │  │ Manager         │<─>│ (sealed data store,   │  │
   register/execute/     │  │ (state machine, │   │  e.g. Gramine         │  │
   delete                │  │  locking)       │   │  Protected Files)     │  │
                         │  └──────┬──────────┘   └───────────────────────┘  │
                         │         │                                         │
                         │         V                                         │
                         │  ┌─────────────────┐   ┌───────────────────────┐  │
                         │  │ WASM Execution  │   │ Attestation /         │  │
                         │  │ Sandbox         │   │ Deletion Proof        │  │
                         │  │ (wazero,        │   │ Module                │  │
                         │  │  network        │   │ (SGX quote            │  │
                         │  │  blocked / FS   │   │  generation,          │  │
                         │  │  restricted)    │   │  certificate signing) │  │
                         │  └─────────────────┘   └───────────────────────┘  │
                         └───────────────────────────────────────────────────┘
```

`wasm_runner` プロセス自体がエンクレーブ内で動作するため、ライフサイクル管理・
封印ストレージ・実行サンドボックス・証明発行のすべてが TEE の保護下にある。
外部との通信経路は HTTP（クライアントとのやり取り）のみであり、WASM 実行中は
ネットワークアクセスを提供しないことで、実行中のプログラムからの外部送信を
物理的に不可能にする（詳細は §8）。

## 5. データのライフサイクル状態遷移

```
                register
  (nonexistent) ────────> REGISTERED ──────────> IN_USE
                              ^                    │
                              └────────────────────┘
                              execute complete
                              │
                              │ delete
                              V
                          DELETING ─────────> DELETED (terminal)
                       (sealed data erasure + deletion certificate issuance)
```

| 状態 | 説明 | 許可される操作 |
|---|---|---|
| `REGISTERED` | データが封印保存され、利用可能な待機状態 | `execute`, `delete`, `status` |
| `IN_USE` | 特定の WASM プログラムを実行中（一時状態） | `status` のみ（`execute`/`delete` は排他） |
| `DELETING` | 削除処理中（封印データの消去・証明生成） | `status` のみ |
| `DELETED` | 削除完了。終端状態、データIDの再利用不可 | `status`, `proof`（削除証明の再取得） |

`execute` は1回の実行で複数のデータIDを同時に指定できる（0個以上の可変長、§7）。
この場合、指定された全データが原子的（all-or-nothing）に `IN_USE` へ遷移し、
実行完了時に全件が `REGISTERED` に戻る。1件でも取得できない場合は、どのデータの
状態も変更しない。

### 不変条件

1. 生値（平文）を直接返す専用 API（例: `GET /data/{id}`）は**一切存在しない**。
   `register` は書き込み専用、`execute` は WASM プログラムの計算結果のみを返す。
   ただし `execute` の結果内容自体は WASM プログラム次第であり、生データがそのまま
   出力される可能性を排除しない（利用者＝データオーナー本人を前提とすることで
   許容している。§8-3, §10 参照）。
2. `IN_USE` 中は同一データIDに対する他の `execute`/`delete` をブロックする
   （データIDごとの排他ロック）。実行途中の状態に対する競合操作
   （TOCTOU: Time-of-check to time-of-use）を防ぐため。
3. `DELETED` は終端状態であり、以後そのデータIDへの `execute` は 404 を返す。
   同じ ID の再登録も禁止する（監査ログとの対応を一意に保つため）。
4. 状態遷移はすべて `wasm_runner` 内の単一の Lifecycle Manager が管理し、
   状態を書き換えられる経路を一本化する。

## 6. コンポーネント設計

### 6.1 Lifecycle Manager

- データIDごとに `{state, storage_ref, owner_id, content_hash, created_at}`
  を保持するメタデータストアと、状態機械の遷移ロジックを持つ。
- データIDごとの排他制御（mutex）により、`IN_USE`/`DELETING` 中の競合操作を防止する。
- メタデータ自体もプロセス再起動で失われてよいものではないため、封印ストレージ側に
  永続化する（§6.2）。

### 6.2 封印ストレージ（Secure Storage）

- 登録されたデータは平文のままディスクに置かず、暗号化して保存する。
- Gramine が提供する **Protected Files**（`sgx.protected_files`）機能の利用を第一候補
  とする。ファイル単位で AES-GCM 相当の暗号化・完全性検証が行われ、鍵はエンクレーブ
  外に露出しない。
- 鍵のプロビジョニング方式（固定鍵 / RA-TLS 経由の動的配布）は §11 の未解決課題とする。
- 削除は物理的なファイル消去に加え、対応する暗号鍵・メタデータを破棄する
  **クリプトシュレッディング**を正とする（§9）。

### 6.3 WASM 実行サンドボックス

- 現行の `handleExecuteWasm`（`wasm_runner/main.go`）をベースに、以下を追加する。
  - 実行対象データを読み取り専用のスコープ付きファイルとして WASM 側にマウント。
  - ネットワーク関連の WASI ホスト関数を一切 instantiate しない。
  - 実行タイムアウト・メモリ上限を設定する。
- 詳細は §8。

### 6.4 Attestation / Deletion Proof Module

- SGX のリモートアテステーション機構（`/dev/attestation/*` 経由の quote 生成）を
  用いて、削除イベントに対する証明を発行する。
- 詳細は §9。

## 7. API 設計

WASM 実行は単一の `POST /execute` に統合し、使用する登録済みデータをクエリ
パラメータ `data` の繰り返しで **0個以上・可変長**で指定する
（`POST /execute?data=<id1>&data=<id2>`。複数データを跨ぐ計算を可能にするため）。
旧 `POST /execute-wasm`（ステートレス実行）はデータ指定0個の場合として、
旧 `POST /data/{id}/execute` はデータ指定1個の場合として、それぞれ包含される。

| メソッド / パス | 説明 |
|---|---|
| `GET /` | ヘルスチェック（既存） |
| `POST /users` | ユーザ発行。応答: `{ owner_id, api_key, created_at }`（API キーの平文はこの応答限り。§7.1） |
| `POST /execute` | WASM 実行。ボディ＝WASM バイナリ。クエリ `data`（繰り返し可）で使用する登録済みデータを0個以上指定（1個以上の指定は認証必須）。応答: 実行結果 |
| `POST /data` | データ登録（認証必須）。ボディ＝データ本体。応答: `{ data_id, registered_at }` |
| `DELETE /data/{id}` | データ削除（認証必須、所有者本人のみ）。応答: 削除証明（JSON） |
| `GET /data/{id}/status` | 現在の状態（`REGISTERED`/`IN_USE`/`DELETING`/`DELETED`）を返す。生データは返さない |
| `GET /data/{id}/proof` | `DELETED` 状態のデータに対し、削除証明を再取得（監査用） |

### 7.1 ユーザ認証と所有者照合

- ユーザは `POST /users` で発行され、恒久的な識別子 **owner_id** と、それを証明する
  秘密 **API キー**の対で表される。サーバは sha256(API キー) → owner_id の対応表
  （ユーザ表）だけを封印ストレージに保存し、キーの平文は発行時の応答限りとする。
- リクエストは API キー（`X-API-Key` または `Authorization: Bearer` ヘッダ）で
  認証され、owner_id に解決される（**認証**）。データレコードには登録者の owner_id
  が記録され、`execute`/`delete` では対象の全データについて認証済みユーザが所有者
  本人であることを照合する（**認可**）。本書の前提「利用者＝データオーナー本人」
  （§3）はこの照合によってシステム的に保証される。
- 秘密（API キー）と識別子（owner_id）を分離しているため、キーが漏洩した場合は
  ユーザ表のエントリ差し替えのみで失効でき、データレコード側は影響を受けない。
  owner_id は秘密ではないため、ログ・削除証明・監査記録に含めてよい。
- `register`・データ指定のある `execute`・`delete` は認証必須。ステートレス実行
  （データ指定なし）と `status`/`proof` は認証不要とする。

`POST /execute` の詳細:

- データ指定0個はステートレス実行（動作確認用）であり、ライフサイクル管理・
  削除証明の対象にはならない点をドキュメント上明記する（登録されていないデータに
  削除証明という概念はそもそも成立しないため）。
- 複数データを指定した場合、指定した全データが実行中 `IN_USE` になる（§5）。
  取得は all-or-nothing とし、1件でも存在しない・削除済み・使用中・所有者
  不一致の場合は、どのデータの状態も変更せずエラーを返す。同一IDの重複指定は
  拒否する。
- 指定順の i 番目のデータが読み取り専用ファイル `/data/input<i>` として WASM から
  見える（§8-2）。
- データを1個以上指定する場合は認証必須であり、指定した全データの owner_id が
  認証済みユーザと一致しなければならない（§7.1）。したがって複数データを跨ぐ
  実行は同一オーナーのデータの組み合わせに限られ、利用者＝データオーナー本人の
  前提（§3）が保たれる。

## 8. WASM プログラムの実行制約

「任意のプログラムを実行できるが、外部送信はできない」を実現するための制約。

1. **ネットワークアクセスの禁止**
   - wazero に WASI Socket 拡張やその他ネットワーク系ホスト関数を一切登録しない。
     現行実装は `wasi_snapshot_preview1` の Core Module 相当のみを instantiate して
     おり、素の状態でもソケット API は提供されていない。これを維持し、将来的にも
     ネットワーク系ホスト関数を追加しないことを設計原則として明文化する。
   - 多層防御として、Gramine のマニフェスト側でも該当プロセスからのネットワーク
     syscall 発行経路を最小化する設定を検討する。

2. **ファイルシステムアクセスの制限**
   - `wazero.NewModuleConfig().WithFS(...)` 等を用い、当該実行で指定された登録
     データ（0個以上、§7）のみを、指定順に読み取り専用で `/data/input0`,
     `/data/input1`, ... としてマウントする。データ指定なしの実行では FS 自体を
     与えない。
   - ホストの他ディレクトリへの経路は一切与えない。書き込み用マウントも提供しない
     （結果は stdout 経由でのみ返す）。

3. **出力サイズの制限**
   - 既存の `cappedBuffer`（`outputCap = 1 MiB`）を維持する。ただしこれはリソース
     枯渇（DoS）防止のための制限であり、機密性の担保を目的としたものではない。
   - stdout に生データがそのまま出力されても、利用者＝データオーナー本人（§2, §3）
     である限り、正当なデータ保有者が自分自身のデータを見ているに過ぎず、
     「意図しない第三者への漏洩」には該当しない。したがって出力内容を検査する
     ポリシーチェック（DLP 的な仕組み）は導入しない。変換されたコピー（暗号化・
     符号化・要約等）は原理的に検出できず不完全な対策にしかならないため（§11）。

4. **リソース制限**
   - 現行実装には実行タイムアウトが存在しないため、`context.WithTimeout` を
     `InstantiateModule` に適用する。
   - `wazero.NewRuntimeConfig().WithMemoryLimitPages(...)` 等でメモリ上限を設定し、
     無限ループやメモリ枯渇によるサイドチャネル・DoS を緩和する。

5. **最小権限の原則**
   - instantiate する WASI ホスト関数は必要最小限に絞る（`args`, `environ`,
     `clock`, `random` 等は用途を精査し、不要であれば無効化する）。

## 9. 削除証明（Deletion Proof）の設計

### 9.1 削除の実体

- 封印ストレージ上の暗号化データを消去するとともに、対応する暗号鍵・メタデータを
  破棄する（クリプトシュレッディング）。暗号化 blob 自体がストレージ上に残存しても、
  復号不能であれば実質的に削除されたとみなす。

### 9.2 削除証明のフォーマット（例）

```json
{
  "data_id": "d-xxxxxxxx",
  "deleted_at": "2026-07-08T12:34:56Z",
  "content_hash": "sha256:...(登録時点で記録した元データのハッシュ)",
  "enclave_report": {
    "mrenclave": "...",
    "mrsigner": "...",
    "quote": "base64:..."
  },
  "signature": "base64:...(エンクレーブが保持する鍵による署名)"
}
```

### 9.3 検証手順（第三者検証者側、参考）

1. `quote` を Intel PCS/DCAP 等で検証し、`mrenclave`/`mrsigner` が期待値と一致する
   ことを確認する。
2. エンクレーブ起動時に一度だけ発行された quote の `user_report_data` に、以後の
   署名に使う公開鍵のハッシュが埋め込まれていることを確認する（毎回 quote を
   生成するコストを避けるため、起動時に一度だけ鍵を証明し、以後はその鍵で署名する
   運用とする）。
3. `signature` を当該公開鍵で検証する。
4. `content_hash` が登録時に取得したハッシュと一致することを確認し、証明対象の
   データが登録されたデータそのものであることを確認する。
5. 以上がすべて成立すれば、「当該データはこの（真正な）エンクレーブ内で削除された」
   ことが第三者に対して証明される。

## 10. 脅威モデルと対策

| 脅威 | 対策 |
|---|---|
| ホスト OS / インフラ管理者によるメモリ盗聴・改ざん | SGX のメモリ暗号化・アクセス制御 |
| 悪意ある WASM プログラムによるネットワーク経由のデータ持ち出し | ネットワーク系ホスト関数を一切提供しない（§8-1） |
| 悪意ある WASM プログラムが stdout 経由でデータをそのまま出力する | 脅威ではない：利用者＝データオーナー本人（§2, §3）を前提とするため、出力を受け取るのは正当なデータ保有者自身。`outputCap`（§8-3）はDoS対策として別途維持 |
| データオーナーと異なる主体が実行結果（≒生データ）を取得する | 設計上非対応（§3対象外）。利用者＝データオーナー本人であることを所有者照合（§7.1）で保証することで回避 |
| API キーの窃取による他人のデータの利用・削除 | キーの平文はサーバに保存しない（ハッシュのみ、§7.1）。通信経路上の窃取への対策（TLS等）は本書の対象外。漏洩時はユーザ表のエントリ差し替えで失効 |
| 削除証明の偽造 | SGX リモートアテステーションによる署名鍵の真正性保証（§9） |
| `IN_USE`/`DELETING` 中の競合操作（TOCTOU） | データIDごとの排他ロックと厳格な状態機械（§5） |
| 削除済みデータのバックアップ等からの復元 | クリプトシュレッディングにより鍵を破棄し、平文的価値を除去（§9.1） |
| SGX サイドチャネル攻撃 | 本書のスコープ外（ハードウェア/Gramine 側の対策に委ねる） |

## 11. 制限事項・未解決課題

- **利用者≠データオーナーへの拡張**: 本書は利用者＝データオーナー本人であることを
  前提とすることで、stdout 経由の情報持ち出し問題（1回の一括コピーおよび繰り返し
  呼び出しによる段階的コピーの双方を含む）を設計上回避している。将来、データ
  オーナーと異なる主体に実行を委譲するユースケース（データマーケットプレイス等）を
  サポートする場合、出力マスキング／DLP的な検査では変換されたコピー（暗号化・
  符号化・要約等）を原理的に検出できず不完全な対策にしかならないため、別の保護
  機構（例: 実行可能な処理を事前承認された関数に限定する、差分プライバシー的な
  出力制約を課す等）の再設計が必要になる。
- **第三者検証者の実装**: 本書はエンクレーブ側での証明**生成**のみを扱う。検証者側
  ツールの実装は別プロジェクトとする。
- **認可設計の残課題**: ユーザ認証（API キー → owner_id）と所有者照合（§7.1）に
  より、利用者＝データオーナー本人の前提（§3, §10）はシステム的に保証されるように
  なった。残る課題は、`POST /users` 自体の保護（誰がユーザを発行できるか）、
  API キーのローテーション・失効の仕組み、権限の粒度（削除のみの委任など）の設計。
- **鍵プロビジョニング**: Gramine Protected Files の暗号鍵をどう配布するか
  （固定鍵 / RA-TLS 経由の動的配布）は要検討。

## 12. 実装計画（マイルストーン案）

1. **Phase 1**: ライフサイクル状態機械 + `register`/`execute`/`delete`/`status` API
   の実装（暗号化なし、まずは状態遷移とロックの正しさを確認）。
2. **Phase 2**: Gramine Protected Files による封印ストレージの導入。
3. **Phase 3**: wazero 実行制約の強化（FS スコープ限定、タイムアウト、メモリ上限）。
4. **Phase 4**: SGX リモートアテステーションによる削除証明の署名・発行機能。

## 13. 現行実装との差分まとめ

- **現状**（`wasm_runner/main.go`）: `POST /execute-wasm` はステートレスであり、
  送信された WASM バイナリをその場で wazero 上で実行し、stdout/stderr を返すのみ。
  データの永続化・ライフサイクルという概念自体が存在しない。
- **拡張後**: データの登録・利用・削除を Lifecycle Manager が状態機械として管理し、
  WASM 実行は wazero の機能（`WithFS`、メモリ上限、タイムアウト等）とネットワーク
  ホスト関数の非提供により厳格に制約される。削除時には SGX リモートアテステーション
  に基づく削除証明が発行される。WASM 実行 API は `POST /execute` に一本化され、
  1回の実行で使用する登録済みデータを可変長（0個以上）で指定できる（§7）。
